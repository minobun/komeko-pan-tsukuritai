# コーディング規約

拡張性が高くシンプルな作りを保つための規約。AIエージェントによる開発を含め、
このリポジトリの変更はすべて本規約に従う（AGENTS.md から参照）。

## 設計指針

SOLID原則を意識し、「シンプルに保ちつつ、変更に強い」構成を優先する。

- **単一責任（S）** — 1モジュール・1コンポーネントは1つの関心事だけを持つ。
  データ取得・判定ロジック・描画を1ファイルに混ぜない（下記「レイヤー分離」参照）。
- **開放閉鎖（O）** — 機能追加は既存コードの書き換えではなく、関数・コンポーネントの
  追加で実現できる形を目指す。条件分岐の増殖より、データ駆動（型・定数・propsで拡張）を選ぶ。
- **リスコフの置換（L）** — 同じ型を受ける関数・コンポーネントは、型が許すすべての値
  （`null` フィールド、空配列等）で正しく動くこと。特定データの決め打ちをしない。
- **インターフェース分離（I）** — コンポーネントには必要な情報だけを渡す。
  巨大なオブジェクトを丸ごと渡さず、`Pick<...>` などで必要なフィールドに絞る
  （例: `RecipeFlour.brand` は `FlourBrand` の一部フィールドのみ）。
- **依存性逆転（D）** — 上位レイヤー（ページ・コンポーネント）は下位の実装詳細
  （Supabaseクライアント等）に直接依存せず、`src/lib/` の関数・型を介して利用する。

過剰な抽象化はしない。「2回目の重複が出てから共通化する」程度を目安に、
将来の拡張を妨げない最小の構造を保つ。

## レイヤー分離

現行の構成を維持する。新しいコードは以下のいずれかの層に置く。

| 層 | 場所 | 責務 |
| --- | --- | --- |
| データ取得 | `src/lib/data.ts` | Supabaseへのクエリを集約。ページから直接DBクエリを書かない（spec.md §7-3）。取得失敗時はフォールバック値を返し、サイト表示を止めない（spec.md §6） |
| 外部クライアント | `src/lib/supabase.ts` | Supabaseクライアントの生成。外部依存のモック境界 |
| 表示用の型 | `src/lib/types.ts` | データモデルに対応する型定義と、型に密接な純粋関数（例: `isGlutenFree`） |
| ロジック | `src/lib/filters.ts` 等 | 絞り込み・変換・判定などの純粋関数。UIから独立させ、ユニットテスト可能にする |
| UI部品 | `src/components/` | 描画とUI都合（URLクエリ同期等）のみ。ビジネスロジックを持たない |
| ページ | `src/app/` | ルーティングとデータ取得の呼び出し、コンポーネントの組み立て |

- 判定・変換などのロジックはUIコンポーネントに直接書かず、`src/lib/` の純粋関数に切り出す。
- 外部依存（Supabase等）はモジュール境界（`src/lib/supabase.ts`）でモックする。

## 命名・コンポーネント分割

- ファイル名はケバブケース（`recipe-card.tsx`）、コンポーネント・型はパスカルケース
  （`RecipeCard`）、関数・変数はキャメルケース（`isGlutenFree`）。
- コンポーネントは名前付きエクスポートとする（`export function RecipeCard(...)`）。
  ページ（`page.tsx` 等）のみNext.jsの規約に従い `default export`。
- データ取得関数は `get<対象>` （`getRecipes`, `getRecipeById`）で統一する。
- 真偽値は `is` / `has` プレフィックス（`isGlutenFree`, `has_gluten`）。
  DB由来のフィールドはDBのスネークケースをそのまま使う。
- 1コンポーネントが複数の関心事（カードの描画とフィルタ状態の管理等）を持ち始めたら
  分割する。繰り返し要素（カード等）と一覧（グリッド等）は別コンポーネントにする
  （例: `recipe-card.tsx` と `recipe-grid.tsx`）。

## 開発フロー：仕様 → テスト → 実装（テストファースト）

このリポジトリでは、機能追加・変更は次の順で進める。

1. **仕様を確認する** — `docs/mvp/spec.md` や該当issueで「何をどう振る舞わせたいか」を先に固める。仕様が曖昧なら実装前にissueで明確化する。
2. **テストを書く** — 期待する振る舞いをユニットテストとして先に書き、失敗することを確認する。
3. **実装する** — テストが通る最小限の実装を行い、リファクタリングする。

バグ修正も同様に、まず再現するテストを書いてから直す。

## テスト

- テストランナーは [Vitest](https://vitest.dev/)（Next.js公式ガイド
  `node_modules/next/dist/docs/01-app/02-guides/testing/vitest.md` に準拠）。
- 実行方法:
  - `npm test` — 全テストを1回実行（CIと同じ）
  - `npm run test:watch` — ウォッチモード
- テストファイルは対象モジュールの近くに `__tests__/<対象名>.test.ts` として置く
  （例: `src/lib/__tests__/filters.test.ts`）。
- PRごとにGitHub Actions（`.github/workflows/test.yml`）でテストが自動実行され、
  失敗するとマージできない。
- `async` Server Componentのユニットテストは現状Vitest非対応のため、ロジックを
  `src/lib/` に切り出してテストする。

## PR運用ルール：PRを小さく保つ

一度の変更で壊れる範囲を小さくし、レビューを速く確実にするため、PRは小さく保つ。

- **1PR = 1つの関心事。** 「ついでのリファクタリング」「無関係なtypo修正」は別PRに分ける。
- **差分サイズの目安は変更 ±300行程度**（生成物・ロックファイルを除く）。
  超えそうなら分割を検討する。ドキュメントのみ・機械的な一括置換はこの限りではない。
- **大きくなる場合の分割の仕方** — 依存の下流から上流へ、それぞれ独立してマージできる
  単位に分ける。典型的な順序:
  1. DBマイグレーション・データ投入（後方互換に注意）
  2. 型・データ取得層（`src/lib/types.ts`, `src/lib/data.ts`）とそのテスト
  3. ロジック（`src/lib/` の純粋関数）とそのテスト
  4. UI（コンポーネント・ページ）
- 分割したPRは同じissueに紐づけ、PR説明に順序・依存関係を書く（例: 「#XXの後にマージ」）。
- ブランチ名は `feature/issue-<番号>-<内容>` とし、PRからissueを参照する。
