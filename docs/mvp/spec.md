# 米粉パンレシピ×米粉銘柄まとめサイト 仕様書

## 1. 背景・課題

- 米粉パンのレシピは既存の大手レシピサイト（楽天レシピ、Nadia、デリッシュキッチン等）に分散しており、米粉に特化した横断検索ができない。
- レシピの材料欄は「米粉 200g」のような記載が多く、**どの銘柄の米粉を使ったか**が不明瞭。
- 米粉は銘柄によって成分（製パン用グルテン入りか、サイリウム同梱か、デンプン損傷度、原材料米の産地等）が異なり、同じレシピでも仕上がりが大きく変わる。
- 初心者はこの違いを知らずに再現に失敗しやすい。
- **逆方向の悩みも大きい**：「家にある米粉で何が作れるか分からない」「レシピ指定の銘柄が手に入らないとき、手元の銘柄で代用できるか分からない」。

## 2. コンセプト

外部レシピ本文はスクレイピング・転載せず、**「レシピへのリンク」と「そのレシピで使われている米粉銘柄」の紐付け情報のみ**を自前で保有・公開するキュレーション型サイト。

差別化の柱は次の2点：

1. **銘柄起点の逆引き**：「この米粉で作れるレシピ一覧」「この銘柄はグルテンフリーレシピに代用可能か」という、既存サイトにない検索軸を提供する。
2. **実食レビューを主コンテンツとする**：運営者が実際に焼いた仕上がりメモ・銘柄間の比較を独自コンテンツとして蓄積する。単なるリンク集ではSEO上「薄いコンテンツ」と評価されやすいため、実食情報がサイトの本体であるという位置づけ。紐付け情報には根拠（銘柄指定あり／銘柄指定なし／目視で確認可能）を明示し、誠実さを差別化要素とする。

## 3. スコープ

### やること
- レシピの外部リンク、掲載サイト名、著作者名、パン種別を登録
- 米粉銘柄のマスタ情報を整備（成分・画像・公式サイト・購入リンク・廃番情報を含む）
- レシピと米粉銘柄の紐付け（1レシピ複数銘柄も許容）、紐付け根拠のステータス付き
- 銘柄別・パン種別・グルテンフリー有無などでの一覧・絞り込み表示
- 銘柄詳細ページ（逆引き：その銘柄で作れるレシピ一覧）
- 初期データはこちらで実食・目視確認のうえ手入力（人力チェックによる編集コンテンツとして著作権面をクリアにする）
- リンク切れの定期チェックと表示制御
- 削除依頼・問い合わせ窓口の設置

### やらないこと（初期スコープ外）
- レシピ本文の自動取得・スクレイピング
- 企業・メーカーとの提携やリアルタイム在庫連携
- ユーザー投稿機能・API公開（将来対応。ただし §7 の設計方針で下地は作る）

## 4. データモデル

全テーブル共通：`id uuid PK` / `created_at timestamptz` / `updated_at timestamptz`

## 4.1 recipes（レシピ）
| カラム          | 型          | 説明                                                     |
| --------------- | ----------- | -------------------------------------------------------- |
| title           | text        | レシピ名                                                 |
| url             | text        | 外部リンク（unique制約）                                 |
| site_name       | text        | 掲載サイト名（例：米粉タイムズ、Nadia）                  |
| author_name     | text        | レシピ著作者名（attribution として表示）                 |
| bread_type_id   | uuid        | FK → bread_types                                         |
| memo            | text        | 補足メモ（食感・失敗しやすいポイント等）                 |
| uses_psyllium   | boolean     | サイリウムを使うか（null = 未確認）                      |
| uses_gluten     | boolean     | グルテンを使うか（null = 未確認）                        |
| uses_oil        | boolean     | 油を使うか（null = 未確認）                              |
| status          | text        | `published` / `link_broken` / `hidden`（削除依頼対応等） |
| last_checked_at | timestamptz | リンク死活チェックの最終実行日時                         |

- サイリウム・グルテンは米粉自体ではなくレシピ側で加えるかが決まる情報のため、`flour_brands.has_gluten` / `has_psyllium`（銘柄自体の成分）とは別に持つ。油はレシピ側のみ。
- 3項目とも nullable とし、`null` を「未確認」として表示する（既存レシピは全項目が未確認から始まる）。

## 4.2 bread_types（パン種別マスタ）※新設
| カラム     | 型   | 説明                                  |
| ---------- | ---- | ------------------------------------- |
| name       | text | 食パン／丸パン／蒸しパン／ベーグル 等 |
| sort_order | int  | 表示順                                |

フリーテキストだと表記ゆれで絞り込みが機能しないためマスタ化。件数が少なければ Postgres enum でも可。

## 4.3 flour_brands（米粉銘柄マスタ）
| カラム               | 型      | 説明                                                     |
| -------------------- | ------- | -------------------------------------------------------- |
| maker_name           | text    | メーカー名                                               |
| product_name         | text    | 商品名                                                   |
| is_bread_use         | boolean | 製パン用かどうか                                         |
| has_gluten           | boolean | グルテン入りかどうか                                     |
| has_psyllium         | boolean | サイリウム（オオバコ）同梱かどうか                       |
| origin               | text    | 原材料米の産地（任意）                                   |
| jan_code             | text    | JANコード（任意。商品同定・将来のデータ連携用）          |
| official_url         | text    | メーカー公式商品ページ（任意）                           |
| image_url            | text    | 商品画像（自前撮影 or メーカー許諾のあるもの）           |
| purchase_url_amazon  | text    | 購入リンク（アフィリエイト、任意）                       |
| purchase_url_rakuten | text    | 購入リンク（アフィリエイト、任意）                       |
| is_discontinued      | boolean | 廃番フラグ（米粉はリニューアル・終売が多い）             |
| note                 | text    | 銘柄解説（特徴・向いているパン等。銘柄詳細ページの本文） |

## 4.4 recipe_flour_map（紐付け）
| カラム              | 型   | 説明                                                   |
| ------------------- | ---- | ------------------------------------------------------ |
| recipe_id           | uuid | FK → recipes                                           |
| flour_brand_id      | uuid | FK → flour_brands                                      |
| link_status         | text | 紐付けの根拠。`brand_specified`（銘柄指定あり）/ `brand_unspecified`（銘柄指定なし）/ `visually_identified`（目視で確認可能） |
| result_memo         | text | 【廃止予定】仕上がりメモ。データは 4.6 reviews へ移行済み。UIが感想表示へ切り替わり次第カラムを削除する |

- (recipe_id, flour_brand_id) に unique 制約
- レシピに記載のない米粉で作った場合も、このテーブル経由で紐付けて感想を蓄積する（`brand_unspecified`）
- `link_status` の各値の意味
  - `brand_specified` … レシピ本文中に銘柄の指定が明記されている
  - `brand_unspecified` … レシピには銘柄の記載がないが、その米粉で作った実績（感想）がある
  - `visually_identified` … レシピの写真・動画等から銘柄が目視で確認できる
- **「実食したか」は紐付けの根拠ではないため `link_status` では表さない。** 実食の事実は
  4.6 reviews の有無で表現する（旧 `verification_status` の `baked` / `visual` を再定義）
- 仕上がりタグは下記 4.5 で正規化

## 4.5 result_tags / recipe_flour_result_tags（仕上がりタグ）※新設
| テーブル                 | カラム         | 説明                                    |
| ------------------------ | -------------- | --------------------------------------- |
| result_tags              | name           | もちもち／ふわふわ／パサつき 等のマスタ |
| recipe_flour_result_tags | map_id, tag_id | 紐付け（多対多）                        |

絞り込み軸として使うためマスタ化。MVPでは `result_memo` のみで始め、タグは後追いでも可。

## 4.6 reviews（感想）※新設
| カラム      | 型   | 説明                                                                 |
| ----------- | ---- | -------------------------------------------------------------------- |
| map_id      | uuid | FK → recipe_flour_map（1つの紐付けに複数の感想を蓄積できる）         |
| body        | text | 感想本文（もちもち感・膨らみ等の所感）                               |
| flour_tips  | text | その米粉で作る際の工夫コメント（吸水調整などのTips、任意）           |
| author_name | text | 投稿者の表示名（当面は「運営者」固定）                               |
| author_type | text | `operator`（運営者）/ `user`（一般ユーザー）。将来のユーザー投稿（#52）への布石 |

- 当面の入力者は運営者のみ（RLSは読み取りのみ公開。書き込みはservice_role経由）
- 旧 `recipe_flour_map.result_memo` のデータはマイグレーションで本テーブルへ移行済み

## 5. 画面構成

1. **トップページ**：サイト概要、新着レシピ、注目銘柄
2. **レシピ一覧ページ**：メーカー／パン種別／グルテンフリーの有無／仕上がりタグで絞り込み
3. **米粉銘柄一覧ページ**：成分（グルテン・サイリウム）での絞り込み
4. **米粉銘柄詳細ページ**（SEOの主戦場）：銘柄解説、成分情報、購入リンク、**その銘柄を使うレシピ一覧（逆引き）**
5. **レシピ詳細ページ**：使用米粉銘柄タグ、紐付け根拠のバッジと実食確認済みバッジ（感想がある紐付けに表示）、メモ、著作者名・掲載サイト名の明記、外部リンクへの遷移ボタン
6. **About／運営ポリシーページ**：独自調査による記載である旨、削除依頼・問い合わせ窓口

検索流入の想定クエリは「ミズホチカラ 食パン レシピ」「（銘柄名） グルテンフリー パン」のような**銘柄名×パン種別**が中心。銘柄詳細ページとその配下の絞り込み一覧を静的URLとして持つ設計にする。

## 6. 技術構成（無料枠前提）

- フロントエンド／API：Next.js（App Router）、ホスティングは Vercel Hobby（無料）
- DB：Supabase Free
- クロテッドクリームマップと同一スタックを流用（Google Maps API部分は不要）

## 無料枠での注意点と対策
- **Supabase Free はアクセスが約1週間ないとプロジェクトが一時停止する**。閲覧をDB直読みにせず、**ISR（ビルド時＋再検証時のみDBアクセス）** にすることで、DBが止まってもサイト自体は表示され続ける構成にする。データ更新は手動再検証（revalidate）で反映。
- 画像は自前撮影分を Supabase Storage か Vercel の public 配下に配置（点数が少ないうちは後者で十分）。
- アクセス解析は無料のプライバシー配慮型（Vercel Analytics 無料枠等）を利用。

## 代替案（さらにシンプルに始める場合）
初期は閲覧のみ・データ手入力・更新低頻度のため、**リポジトリ内の JSON/YAML → SSG のみ**でも成立する（DB不要・完全無料・休止リスクなし）。ただし将来の投稿機能・API公開を確定的に見据えるなら、最初から Supabase＋ISR 構成を推奨（本仕様はこちらを採用）。

## 7. 将来のAPI公開・投稿機能を見据えた設計方針

初期はAPIを公開しないが、将来 OIDC による外部IdP認証（Google等）を使った投稿機能とAPI公開を想定。今から仕込んでおくこと：

1. **主キーはUUID**（連番だとAPI公開時に総件数・登録順が露出する）— 採用済み
2. **RLSを最初から有効化**：閲覧は匿名read可、書き込みは管理者ロールのみ。投稿機能追加時はポリシー追加で対応できる
3. **読み取りロジックをAPI層に集約**：ページから直接DBクエリを書かず、Next.js Route Handlers（または server 関数）にデータ取得を寄せておくと、将来そのまま公開APIのエンドポイントに転用できる
4. **認証はSupabase Authを想定**：Google等のOIDC/OAuthプロバイダーを標準サポートしており、追加実装コストが小さい
5. **投稿機能はモデレーション前提で設計**：ユーザー投稿URLはスパム・不正アフィリエイトリンクの温床になるため、`draft → pending_review → published` の承認フローを最初からstatus設計に織り込む（4.1 の status はこの布石）
6. **API公開時の必須要件（将来）**：レートリミット、APIバージョニング（/v1/）、利用規約

## 8. 法務・運用

- **著作権方針**：レシピ本文・写真は一切転載しない。リンク＋事実情報（使用銘柄）＋独自の実食メモのみ。各所に「独自調査・実食確認による記載」と注記
- **attribution**：レシピ詳細ページに著作者名・掲載サイト名を明記
- **削除依頼窓口**：問い合わせフォーム（Googleフォームで可）を最初から設置し、依頼があれば `status = hidden` で即時非公開にできる運用にする
- **リンク切れ監視**：GitHub Actions の定期実行（週1程度）で全URLの死活チェック → `link_broken` に更新し、一覧から除外 or 注記表示
- **商標**：メーカー名・商品名は事実の参照として使用し、メーカー公認・提携を装う表現はしない
- **商品画像**：メーカーサイトからの無断転載はしない。自前撮影 or 許諾のあるもののみ

## 9. 収益化（任意・小規模）

- 銘柄詳細ページに Amazon アソシエイト／楽天アフィリエイトの購入リンクを設置
- 「その米粉をすぐ買える」というユーザー価値と両立する自然な導線
- アフィリエイト利用時はプライバシーポリシー・広告表記を About ページに記載

## 10. MVPの進め方

1. Supabase に §4 のテーブルを作成、RLS設定（匿名read／管理者write）
2. 自身が実際に焼いたレシピ・使用銘柄を種データとして5〜10件登録（`link_status` + 感想付き）
3. 既存の人気レシピサイトから、材料欄を目視確認しながら20〜30件を手入力で追加（`visual`）
4. 銘柄マスタは主要10〜15銘柄（ミズホチカラ、共立食品、波里、富澤商店 等）から整備し、銘柄解説を書く
5. レシピ一覧／銘柄一覧／銘柄詳細（逆引き）の閲覧機能でリリース。About・問い合わせ窓口を同時公開
6. GitHub Actions のリンクチェックを設定
7. 反応を見てユーザー投稿機能・API公開・評価タグの拡充を検討

## 11. 今後の検討事項

- コンテンツ収集をユーザー投稿型に広げるか、編集メディア型のまま維持するか（投稿型に広げる場合はモデレーション運用コストを見積もる）
- 米粉銘柄マスタの継続的なメンテナンス方法（新商品・リニューアル・廃番の追従）
- 銘柄間の「代用可否」情報の持ち方（例：グルテンフリー×サイリウム入り同士は代用しやすい、等のルール化 or 個別記載）
- SEO：銘柄名・パン種別での検索流入の実測と、構造化データ（Product / ItemList）の付与
- 実食レビューコンテンツ（同一レシピの銘柄比較記事）をブログ形式で拡充するか
